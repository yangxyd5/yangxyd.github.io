web框架原理之wsgiref和webob

发表于 2017-10-05 | 分类于 Python

WSGI

  1. WSGI (Web 服务器网关接口) 是 python 中所定义的 Web 服务器和 Web 应用程序之间或框架之间的接口标准
  2. WSGI 接口规范的目的就是规范 Web 服务器与 Web 应用之间的交互，在协议之间进行转换
  3. WSGI 将 Web 组件分成三类：Web 服务器、Web中间件、Web应用程序
    • 服务器：指的是实现了调用应用的部分
    • 中间件：处于服务器和应用两侧，起粘合作用，具体包括：请求处理、environ 处理
    • 应用：是一个可调用对象，一般指的是包含一个 __call__ 方法的对象

wsgiref

wsgiref 是 WSGI 细则的一个参考实现，用于给web服务器和框架提供 WSGI 支持。它提供了下面几个子类：

字类 功能
wsgiref.util WSGI 环境工具
wsgiref.headers WSGI response 头工具
wsgiref.handlers WSGI 服务器基类
wsgiref.simple_server 一个简单的 WSGI HTTP 服务器
wsgiref.validate 检测WSGI服务器是否和 WSGI PEP 3333细则 一致的检查器

wsgiref.simple_server

​ 本模块实现了一个简单HTTP服务器，可以为 WSGI 应用提供服务，每个服务器实例在给定的主机和端口上服务一个 WSGI 应用程序，如果你想在单个主机和端口上提供多个 WSGI 应用程序，则应该创建一个 WSGI 应用程序，并且解析 PATH_INFO（客户端请求的地址），以为每个请求分配不同的应用程序，例如：使用来自 wsgiref.util 的 shift_path_info() 函数。

  • 创建服务器

make_server(host,port,app,server_class=WSGIServer,handle_class=WSGIRequestHandler)

创建一个新的监听在 host + port 的 WSGI 服务器，为 app 接受新的连接，返回值是 server_class 的实例，并使用 handle_class 处理请求，其中 app 必须是 PEP 3333 定义的 WSGI 应用程序对象。

  • 一个 WSGI 应用 demo

demo_app(environ, start_response)

这个函数是个小而完整的 WSGI 应用，它能返回 hello world 和 environ 字典内容 组成的文本页面。

  • 细节
  • 请求信息

environ 是一个包含了HTTP请求信息的字典，下面列举一些信息

键 值含义
REQUEST_METHOD 客户端的请求方法
PATH_INFO 请求的URL地址的后半部分
QUERY_STRING 查询字符串，即 ? 后的内容
SERVER_NAME 服务器主机名
SERVER_PORT 服务器的端口
HTTP_HOST 客户端请求头的 HOST 字段
SERVER_PROTOCOL 客户端发送请求所使用的协议版本
CONTENT_TYPE HTTP 请求中任何 Content-Type 域的内容，可能为空或不存在。
HTTP_USER_AGENT 客户端所使用的浏览器信息
  • 响应头

start_response(status, headers, exc_info=None)

start_response，是返回给客户端的响应头部信息，这是一个可调用对象，必须在 WSGI 应用（app）返回前调用，它接受3个参数：

参数 含义
status 响应的状态码，例如：”200 OK”
headers 响应包的HTTP首部信息，每个首部信息的K/V组成一个二元组，放入一个列表中，见：demo_app
exc_info 处理错误的时候使用

解析environ

通过解析：environ['PATH_INFO]，就可以判断用户访问的地址，而使用不同的函数处理响应。

  • 方法1，在APP内处理URL和页面的对应关系
  • 方法2，在一个字典内部处理URL和页面的对应关系，使用 MVC 模式
  1. 将处理的模块放入 Controller 目录的 page.py下
  1. 将 page.py 需要的页面文件放入到 View 目录下的 index.html
  1. 将 page.py 需要的数据库连接等代码 放在 Model 目录中

  • MVC 和 MTV

MVC模式

将数据库的操作代码放在 Model 目录，将模板页面文件放在 View目录，将业务处理逻辑代码放在 Controller目录这样的分离结构组成整个业务系统即由：Model、View、Controller 组成就是MVC模式。

MTV模式

模式就是：Model（数据库）、Template（模板文件）、View（业务处理） django就是MTV的模式

webob.dec

webob.dec 更加简化 response 过程，无需再写 response(environ,start_response) 这个语句了。

被 wsgify 装饰器装饰过的 WSGI APP（可调用对象）需要接收一个 request 对象，然后在函数内可以调用 request 对象写自己的业务逻辑，最后返回结果的时候无需再使用 start_response了，因为 webob.dec 内部的Response 已经替我们做了。
